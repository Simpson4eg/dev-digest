# i18n (next-intl)

The studio uses `next-intl` for translation and locale routing.

## Layout

```
src/i18n/                 ← next-intl config (router, request handler)
messages/<locale>/*.json  ← message catalogs, one file per namespace
```

## Conventions

- **Keys are dot-paths** scoped to the file: `findings.severity.critical`.
- **One namespace per feature** (`findings.json`, `agents.json`, …) — don't dump
  everything into `common.json`.
- **Fallback locale** is English. A missing key in a non-default locale falls
  back to English; a missing key in English is a bug.
- **No string concatenation** for translations — use ICU placeholders:
  `'You have {count, plural, one {# finding} other {# findings}}'`.

## Using a message

```tsx
import { useTranslations } from 'next-intl';

export function FindingsHeader() {
  const t = useTranslations('findings');
  return <h2>{t('header.title')}</h2>;
}
```

In Server Components, use `getTranslations` instead of `useTranslations`.

## Adding a locale

1. Create `messages/<locale>/<namespace>.json` for each namespace.
2. Register the locale in `src/i18n/` config.
3. Run `pnpm typecheck` — missing keys surface as type errors if message
   modules are typed.
